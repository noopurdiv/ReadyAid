# ReadyAid - AI First-Aid Assistant

ReadyAid is a mobile-first emergency support app built during a hackathon.  
It combines an Android app (Jetpack Compose) with a Python RAG backend (FastAPI + ChromaDB + Ollama) to provide fast, contextual first-aid guidance.

## Project Overview

ReadyAid helps users in urgent situations with:

- **Emergency SOS flow** with countdown and quick dialer handoff
- **AI first-aid chatbot** powered by retrieval + local LLM streaming
- **Personalized responses** using saved medical profile data
- **Emergency guide browsing** for offline-style reference flows
- **Voice-assisted navigation and intent handling**

The app is designed for quick action: minimal taps, clear UX, and first-aid-only response behavior.

## Claude Hackathon Alignment

### Selected Track

**Track 1: Biology & Physical Health**

ReadyAid directly addresses emergency first-aid access, health literacy, and triage support in plain language for people who need immediate, understandable guidance.

### Who This Is For (and Why)

- People facing urgent first-aid situations before professional help arrives
- Families/caregivers who need calm, step-by-step instructions
- Users with specific medical context (conditions, allergies, medications) needing safer guidance

This matters because emergency moments are high-stress and medical information is often inaccessible, jargon-heavy, or hard to apply quickly.

### The 3 Required Hackathon Questions

#### 1) Who are you building this for and why do they need it?

ReadyAid is built for everyday people in emergency or near-emergency scenarios who need immediate first-aid support. They need rapid, understandable, and context-aware instructions before reaching formal care.

#### 2) What could go wrong and what would you do about it?

Main risks and safeguards:

- **Risk: Incorrect or unsafe advice**
  - Safeguards: retrieval-grounded responses, first-aid-focused prompting, and explicit emergency escalation language.
- **Risk: Over-reliance on AI instead of professionals**
  - Safeguards: clear disclaimer, emergency intent handling, and SOS flow that directs users to real emergency services.
- **Risk: Sensitive health data misuse**
  - Safeguards: profile data is local on-device (Room), minimal required payload to backend.
- **Risk: Non-medical misuse / prompt drift**
  - Safeguards: backend non-medical query blocking and first-aid-only behavior.
- **Risk: Automation that removes human control**
  - Safeguards: SOS uses dialer handoff (`ACTION_DIAL`) so user remains in control.

#### 3) How does this help people rather than make decisions for them?

ReadyAid provides guidance, not autonomous decisions. Users remain the decision-makers:

- AI explains next steps clearly and transparently.
- SOS flow supports action but does not silently place/terminate calls.
- Recommendations are assistive, with escalation to professional care when needed.

### Ethical Thinking (Explicit)

- **Human agency first:** the app supports, not replaces, user judgment.
- **Transparency:** it states scope and limitations (first-aid assistant, not doctor replacement).
- **Safety boundaries:** highlights when emergency services should be contacted.
- **Data sensitivity awareness:** handles personal profile context with minimal exposure.
- **Failure-aware design:** connection errors and startup issues are surfaced with actionable guidance.

## Tech Stack

### Android App (`ReadyAid/`)

- Kotlin + Jetpack Compose
- Navigation Compose
- Hilt (DI)
- Room (local persistence for profile/chat)
- Coroutines + Flow
- Voice input + intent mapping

### Backend (`server.py`)

- FastAPI + Uvicorn
- ChromaDB vector retrieval
- SentenceTransformers (`thenlper/gte-small`) for embeddings
- Ollama (`phi3:mini`) for response generation
- NDJSON streaming responses (`application/x-ndjson`)

## End-to-End Flow

1. **User opens app**
   - App state loads from Room (`Loading -> Onboarding -> Disclaimer -> Home`).

2. **Profile setup**
   - User enters personal + medical + emergency contact info.
   - Profile is saved in local Room DB.

3. **Chat query**
   - `ChatScreen` sends query and profile payload to backend via `ServerRagClient`.
   - Backend endpoint: `POST /ask`.

4. **Backend processing**
   - Detects condition keyword (burn, bleeding, choking, etc.).
   - Retrieves relevant chunks from ChromaDB (except critical direct-emergency conditions).
   - Builds safety-focused prompt with profile context.
   - Streams answer chunks as NDJSON lines.

5. **Live UI streaming**
   - Android parses each line into `RagResponse`.
   - Chat bubble updates incrementally in real time.
   - Final chunk marks `is_finished = true` and message is persisted.

6. **SOS flow**
   - Countdown starts from SOS button.
   - App opens dialer (`ACTION_DIAL`) with emergency number to keep user in control.

- **Medical-only guardrail** in backend for non-first-aid prompts

## Setup Instructions

## 1) Clone repository

```bash
git clone https://github.com/<your-username>/Claude-IU-Hackathon.git
cd "Claude IU hackathon new"
```

## 2) Backend setup (Python)

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Optional (first time only): build vector index from PDFs

```bash
python ingest.py
```

Start backend:

```bash
uvicorn server:app --host 0.0.0.0 --port 8000 --reload
```

## 3) Android setup

- Open `ReadyAid/` in Android Studio.
- Sync Gradle.
- Run on emulator/device.

### Backend URL config

In `ReadyAid/app/build.gradle.kts`:

- Emulator:
  - `RAG_BASE_URL = "http://10.0.2.2:8000"`
- Physical phone (same Wi-Fi as laptop):
  - `RAG_BASE_URL = "http://<YOUR_LAPTOP_IP>:8000"`

Then **Sync Project with Gradle Files**.

## API Contract

## `POST /ask`

Request body:

```json
{
  "query": "How to treat a burn?",
  "profile": {
    "age": 24,
    "conditions": ["asthma"],
    "allergies": ["penicillin"],
    "medications": ["salbutamol"],
    "medical_history": ""
  }
}
```

Streaming response lines (NDJSON):

```json
{
  "response": "Step text...",
  "sources": ["RedCross"],
  "is_first_aid": true,
  "condition_detected": "burns",
  "is_finished": false
}
```

Final line sets `"is_finished": true`.

## Important Notes / Known Constraints

- The app intentionally focuses on first-aid context and can reject unrelated queries.
- SOS now uses dialer handoff (`ACTION_DIAL`) rather than forcing direct call termination behavior.
- Emulator storage must have enough free host disk space, otherwise AVD can fail to start.

## Troubleshooting

### `Address already in use` (port 8000)

```bash
lsof -ti :8000 | xargs kill -9
uvicorn server:app --host 0.0.0.0 --port 8000 --reload
```

### Android app cannot connect to backend

- Ensure backend is running on `8000`.
- Verify `RAG_BASE_URL` matches your environment.
- Emulator must use `10.0.2.2`, not `localhost`.

### Emulator startup failure (`Not enough space to create userdata partition`)

- Free host disk space.
- Wipe Data / recreate AVD.
